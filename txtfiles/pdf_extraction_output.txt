 1 Introduction  
1.1 Problem Statement 
Global emergency services are facing critical staffing shortages, with reports showing an 
82% deficit in 911 call centers. Human operators are burdened by high call volumes and 
high-stress environments, leading to: 
Increased response times due to cognitive overload. 
Human errors in information recording and dispatch decisions. 
Lack of context, as traditional systems rely on audio-only communication without 
automated sentiment analysis. 
 
1.2    Product Position Statement: 
The Emotion-Aware Emergency Dispatch System (EAEDS) is an AI -powered dispatch 
operator which is constructed for essential emergency response centers. Not like a  
traditional CAD (Computer Aided Dispatch) systems, EAEDS reviews the audio and 
emotion of the caller in real-time with the use of local GPU processing, which provides 
controllers with immediate transcription, emotional telemetry, and triage 
recommendations without depending on external cloud services. 
   
1.3 Project Motivation and Background  
Emergency dispatch systems are considered as the first line of defense in safety of public. 
However, current setups are no longer used, depending a much on manual entry and 
human reasoning under too much pressure. They are encouraged by nowadays 
improvements and updates in local AI inference (Quantized LLMs), the main aim of this 
project is to fill the gap with the help of creating a smart operator that usually listens, 
understands, and thinks at the same time as the human operator and convert raw audio 
into actionable logistical data immediately within a seconds. 
 
1.4 Objectives 
Reduce Triage Time: Automate the collection of details (Name, Location, Type of 
Emergency) to speed up dispatch. 
Enhance Situational Awareness: Provide operators with live visual indicators of the 
caller’s emotional state (Panic vs. Calm). 
Ensure Data Privacy & Reliability: Operate entirely on local hardware (NVIDIA 
GPU), eliminating privacy concerns and internet dependency regarding cloud APIs.  
Operational Resilience: Ensure the interface remains responsive even under heavy load.

1.5 Literature Review and GAP Analysis 
Reddy, S., & Kompalli, S. (2022). Emotion Detection using Multimodal Analysis. - 
Discusses using audio and text for emotion detection but relies on cloud APIs which 
introduces latency. 
Huang, J., et al. (2020). Whisper and ASR Systems. - Establishes the accuracy of 
Whisper but highlights the computational cost for real-time usage. 
 
GAP Analysis: 
Most nowadays existing "Smart 911" solutions depends on cloud-based APIs 
(Google Cloud Speech, AWS Transcribe). This gives introducion to two critical 
gaps: 
 Latency: Internet round-trip times are not acceptable for split -second 
emergency decisions. 
Privacy/Security: Sending emergency call audio to third -party clouds goes 
against the  harsh data sovereignty laws. 
Our Solution (EAEDS) closes these gaps by processing the entire AI pipeline 
(Whisper + BERT + LLM) locally on the workstation's GPU, by making sure 0-hop 
privacy and minimum latency.

2.5 Project Risks and Risk Mitigation Plan  
Project Risks: 
 
GPU Resource Exhaustion:Processing Whisper, LLM, and BERT simultaneously might 
crash VRAM. 
 Model Hallucination:The LLM will might suggest "Fire Dept" for a "Firearm" incident.  
Audio Quality:Poor signal Quality will lead to bad transcription.  
 
Risk Mitigation Plan: 
 
Quantization: 
Usage of 4-bit quantized models (GGUF/AWQ) to fit 3 distinct models into 8GB VRAM.  
Human-in-the-Loop: 
The actions are only recommended by the AI dispatcher, the human Dispatcher must click 
"Dispatch" to confirm it  
Fallback Mode: 
The system reverts to a standard "Dumb" phone interface immediately if the AI fails.

the design methodology in this project and it matches with the structured and recyclable nature 
of the software components.

Project Overview 
The conventional 911 response system is usually improved bu the EAEDS which is commonly 
also called as DispatchAI. It helps in handling the worldwide problem of call centers which is 
understaffed by 82% by adding up an AI assistant which performs its work alongside human 
programmers. The system can listen, understand, and think with the programmers . It works by 
using tools like Whisper, OpenSMILE, BERT, and a fine -tuned language model to change 
audio into applicable data, which includes transcripts, sentim ent analysis, and location 
information. 
 
 
   
 
   
 
 
Scope 
What the System Will Do:  
This project’s range includes the design and creation of functional prototype for an 
Autonomous Dispatch Assistant. The main features of this include:   
Real-time Speech-to-text: Instantly changes caller speech into text using Whisper model.   
Emotion Detection: Inspecting vocal patterns with OpenSMILE to detect emotional states 
such as panic, fear, or neutrality.   
Smart Decision-Making: Using fine-tuned LLMs to follow standard 911 processes (Medical, 
Fire, Police) and produce questions and suggestion according to current scenario of caller.   
Operator Dashboard: A web-based interface (built with Next.js) that shows live transcripts, 
detected emotions, and recommended actions for dispatchers and call history.  
Local GPU  Architecture: A unified system where all AI processing and user interface 
components run on the local system equipped with NVIDIA GPU hardware, eliminating cloud 
dependencies and reducing latency..   
 
What the System Will Not Do:  
The following items are clearly removed from this phase of the project:   
PSTN/Telephony Integration: Currently, the system uses calls using WebSockets and does 
not connect to original landline or mobile networks.   
Legal/Medical Liability: The system only provides recommendations and is not meant to 
make final legal or medical decisions without a human checking them.   
Physical Hardware Deployment: This project is software based only and does not cover 
giving physical servers or specialized radio equipment.   
Multi-Language Support: Our project only works with english language not with other 
languages so it is not multi language project.

High-fidelity real-time audio streams is controlled by the system. Audio is recorded through 
WebSockets, and it was similar to 16kHz mono, and stored temporarily to make sure 
alignment with Whisper (ASR) and OpenSMILE (Acoustic Emotion) models. Transcription 
data is made safe and organised before it is analysed by the LLM for response generation. 
ML Model Integrated Design:  
The architecture combines many professional models: 
Whisper: Provide accurate speech to text.  
BERT + OpenSMILE: It is used for both meaning-based and sound -based emotion 
classification. 
Llama 3/Mistral: It is used for advanced and responsive generation.  
The design focus on low lag and understandability , and present emotion measures and 
provide suggestions to dispatchers in a Human in the loop format.

Maintenance and Evolution:  
The modular design give permission for hot swapping models. The AI service interface make

Risks and Volatile Areas:  
Latency Risks: 
Processing large audio chunks on local GPU may cause latency if hardware is insufficient. 
Contingency: Implementation of batch processing optimization, model quantization, and 
hardware requirement validation to ensure response time under 2 seconds.  
 Model Accuracy: It is the detection of emotions during high stress, noisy environments 
might be sensitive . 
• Contingency: Using a double engine approach like Acoustic along with Semantic for 
verification of indicators. 
 Scaling Volatility:  
Handling multiple simultaneous high bitrate audio streams might strain local GPU VRAM 
and system memory. Contingency: Implementation of queue-based processing and GPU 
memory management with maximum concurrent call limits based on available VRAM. 
• Contingency: Usage of local streaming and horizontal scaling of AI worker nodes.  
 
 
 
System Architecture : 
 

   
 
   
 
High-Level Decomposition:  
The system is divided in three large subsystems: 
Frontend Dashboard: It is the real time visualization and user interaction. 
Backend Orchestrator: It is the routing , management of session and data persistence. 
AI Engine: It is heavy audio working through computation and reasoning. 
Physical Design: 
Execution: The entire system including Frontend, Backend, and AI Engine runs on a single 
local workstation (Windows/Linux) equipped with NVIDIA GPU . 
GPU Layer: All AI models (Whisper, BERT, OpenSMILE, LLM) execute on the local 
NVIDIA GPU (minimum 6GB VRAM recommended, 8GB+ for optimal performance) .  
Error Handling: Disconnection are handled by complete try except blocks in the WebSocket 
loop  and partial transcripts are saved by complete try except blocks in the WebSocket loop 
which is done to prevent data loss.  
Software Architecture :  
The system comes after a simple Three-Tier Architecture: 
User Interface Layer:It is mainly constructed with Next.js, providing the "Phone UI" for 
callers and "Midnight Dashboard" for users. 
Middle Tier (Business Logic): FastAPI works by controlling WebSocket bi-directional 
streaming, orchestrating AI service calls, and controlling local business rules. 
Data Access Layer: SQLite operations are abstracted by database manager class for 
protecting and retrieving call logs. 
GPU Resource Management Layer: PyTorch/CUDA interface manages model loading, 
inference batching, and VRAM allocation on the local GPU to ensure optimal performance 
across all AI components. 
 
   
 
   
 
 
Design Strategy: 
• Future Extension: It works by using an Abstract Base Class AIService which helps 
to allow simple integration of new LLMs or ASR models. 
• System Reuse: Components like ConnectionManager are constructed for any real-
time broadcasting requirement . 
• Data Management: It works by using JSON strings in SQLite to save complex 
transcript structures. 
• Concurrency: It uses Python's async/await and FastAPI's Websocket abilities to 
control simultaneous connections without any delay.

Overall System Description:  
Project Background: 
Emergency dispatch system are the first and the most essential line of defense in safety of 
public. However the current setups are mostly old, depending on physical work and human 
reasoning during hard times. The Emotion-Aware Emergency Dispatch System (EAEDS), 
also called dispatch Al, was made to enhance this procedure. By using nowadays 
advancements in Large Language Models (LLMs) and sound investigation the project aims to 
develop a smart dispatch helper. This helper can listen, understand and think alon g with 
human operators, changing raw audio into usable information. 
 
   
 
 
 
 
Problem Statement: 
Global emergency services are facing a lot of staffing problems, with reports showing 82% 
lack in 911 call centers. Human operators are burdened by high call sounds, which causes:  
Increased Response Times: Postpones in managing information and sending units.  
Human Error: High-stress environment tends to mismanagement or mental exhaustion.  
Lack of context: Traditional systems depends on conversation with automatic devices to 
check call emotion straightforwardly. There is a crucial need for networking layers which can 
manage Early screening, record calls correctly and give decision support without changing 
human element.  
Project Scope:  
This project’s range includes the design and creation of functional prototype for an 
Autonomous Dispatch Assistant. The main features of this include:  
Real-time Speech-to-text: Instantly changes caller speech into text using Whisper model.  
Emotion Detection: Inspecting vocal patterns with OpenSMILE to detect emotional states 
such as panic, fear, or neutrality.  
Smart Decision-Making: Using fine-tuned LLMs to follow standard 911 processes 
(Medical, Fire, Police) and produce questions and suggestion according to current scenario of 
caller.  
Operator Dashboard: It is a web based interface that is developed with Next.js and it 
represents live transcripts, detected emotions, and recommended actions for operators and 
call history. 
Local GPU Architecture: A unified system where all AI processing and user interface 
components run on the local workstation equipped with NVIDIA GPU, providing faster 
response times and eliminating internet dependency for core operations.

Non-Functional Requirements: 
Performance Requirements: 
Speech Transcription Latency: The system must transcribe audio as soon as it is received, 
and it needs to do this within 2 seconds to keep up with the event as it happens.  
Emotion Inference Speed: BERT and Open Smile will identify emotions from the audio and 
send the results to the dashboard within 3 seconds of receiving the audio.This allows 
dispatchers to assess the caller's distress level in real time. 
Safety Requirements: 
Fail Safe Mechanism: The dispatcher will handle the call without any service interruption, if 
the system breaks down or there is any delay or problem in the AI pipeline. The system will 
be changed to Manual Dispatch Mode, when the AI services are not available. 
Right Location Verification: Every reverse geocoded address must be tested against a 
trusted mapping service like the Map Box. This will make sure that first responders are 
scheduled to the real and original location with an accurate measurement of or within 50 
meters of the recorded location. 
Security Requirements: 
Encryption At Rest: The database must be protected using strong encryption, such as AES-
256, to keep caller information private.Important call recordings and transcripts should be 
stored in an encrypted database.  
Authentication: "Midnight Command" dashboard is protected by a JSON Web Token 
(JWT).Session tokens will be expired after 12hours of usage.. 
VoIP Security: To prevent unauthorized access to emergency conversations, SIP signaling 
and RTP media streams will be secured using TLS and SRTP, respectively. 
Reliability Requirements: 
System Availability: The main models and telephone parts need to be available 99% of the 
time to ensure the line is ready for emergency calls. 
Automatic Recovery: A process control system (Docker) should manage the Fats API 
backend (Docker) which can restart its  working procedures in 5 seconds if they are not 
working or they stop operating . 
   
 
 
Usability Requirements: 
UI Design: The “Midnight Command” dashboard must have a dark mode color scheme that 
is present with clear alert presenters which will  help to reduce eye strain during long and late 
night shifts. 
Single Critical Page: The interface should must represent the live location on a map, along 
with a live transcript and emotion graphs. They all should be on one screen so operators can 
easily have access to significant situation details and can decide accurately and can make 
better decisions. 
Supportability Requirements: 
Modular Architecture: A loosely connected Microservice architecture (FastAPI) should be 
used to built the system so that individual AI models, like switching from BERT to Roberta, 
can be improved or changed without the need of reconstruction of the whole system or can 
cause any issue or downtime. 
Deployment Consistency: Our project have to deploy and use docker containers to make 
sure that development,testing and production environment are same and has no overlapping 
or conflicts. 
User Documentation: 
The following documentation must be included with each system release: 
Dispatcher User Manual: A guide that explains all the features used in main midnight 
dashboard and explain overall features of our project. 
System Administration Guide: Technical instructions for handling database backups and
